# ecm-research-case-search

## 定位与边界

本 skill **负责**：
- 对用户给定的研究主题进行**多来源案例检索**（裁判文书 / 证监会处罚 / 交易所纪律处分 /
  并购重组委审核 / 上市委问询与审议）
- 按 [shared/templates/research-output-format.md](../../shared/templates/research-output-format.md)
  的**统一格式**输出案例卡片、可靠性等级标注和适用性分析
- 主动提示检索局限（数据源覆盖缺口、时效性、关键词可能遗漏）
- 在外部检索不可用时，按 fallback 规则降级并**明确标注**

本 skill **不负责**：
- 做法规条文检索（由 `ecm-research-reg-search` 做）
- 做法规效力层级和冲突解决分析（由 `ecm-research-reg-study` 做）
- 对案例结论直接形成法律意见（案例是论据，法律意见由签字律师作出）
- 生成实质性方案建议（由 `ecm-design-*` 系列做）

## 免责声明

本 skill 产出的案例检索结果仅为律师研究的辅助工具。检索覆盖范围、时效性、来源可靠性均有
局限；律师在作为意见书或尽调报告论据使用前，**必须亲自到官方数据源核对原文**。完整免责声明见仓库
顶层 [DISCLAIMER.md](../../DISCLAIMER.md)。

## 前置依赖

- 输出格式：[shared/templates/research-output-format.md](../../shared/templates/research-output-format.md)（统一输出契约，不得在本 skill 另定义格式）
- 外部能力：WebSearch（优先）、WebFetch（针对单一判决书 URL 提取全文）——均为 Claude 环境内置
- 外部 skill：无硬依赖。读 PDF 判决书需调用 `pdf` 外部 skill 时，本 skill 发起协同即可，不在 frontmatter 声明硬依赖

## 核心任务

收到用户的案例检索请求后，按 **4 个 Phase** 顺序执行：

1. **Phase 1 — 澄清**：最多 3 轮追问确认检索主题、范围、时间窗、筛选条件
2. **Phase 2 — 策略制定**：输出检索关键词组合和来源优先级，供用户确认（可跳过，若请求已足够明确）
3. **Phase 3 — 执行检索**：按策略调用 WebSearch / WebFetch / 用户上传文件
4. **Phase 4 — 输出**：按统一输出格式，生成结构化的案例汇总

---

## Phase 1 — 澄清追问

解析用户输入，优先映射以下字段；缺失且影响重大的字段才追问：

| 字段 | 重要性 | 常见取值 | 缺失时的默认处理 |
|------|-------|---------|----------------|
| 研究主题 | 必填 | 自由文本（如"对赌协议效力"、"同业竞争认定"） | 必问 |
| 检索来源 | 高 | 裁判文书 / 证监会处罚 / 交易所纪律处分 / 并购重组委 / 上市委问询 / 全部 | 默认"全部"，执行时按优先级分阶段检索 |
| 时间范围 | 中 | 如"近五年"、"2020 年以来"、"注册制改革以来" | 默认"近五年"（当前日期往前推 5 年） |
| 板块/行业筛选 | 中 | 主板/科创板/创业板/北交所/全部；制造业/医药/软件等 | 默认"全部" |
| 关联项目背景 | 低 | 用户当前在做的 IPO/并购项目简要描述 | 缺失不追问 |

**硬性限制**：追问不得超过 3 轮。信息不全时在输出的"0. 检索/研究策略"里标注"（未澄清，按默认值执行）"并继续。

### 追问示例

- 主题模糊：
  > "您希望检索的是'对赌协议效力'的**哪一类**争点？——例如 (a) 对赌条款本身是否有效 / (b) 与公司对赌 vs. 与股东对赌 / (c) 股权回购权行使条件 / (d) 业绩承诺补偿计算方式 / (e) 全部。"
- 来源不定：
  > "是否要同时覆盖司法案例（裁判文书）和监管案例（证监会处罚、交易所纪律处分、上市委问询）？
  > 如仅关心监管口径，可只选后三类。"
- 时间范围不定：
  > "是否限定检索时间范围？默认近五年，如需看更早的先例（如最高法指导性案例），可扩展至十年。"

## Phase 2 — 检索策略

在开始实际检索前，向用户**同步**检索策略，允许用户在执行前调整。策略包含：

### 2.1 关键词矩阵

**基础关键词**：主题本身 + 同义词 + 英文对应词。

示例（主题"对赌协议效力"）：
```
基础词：对赌协议 OR 对赌条款 OR 估值调整协议 OR 估值调整机制 OR VAM OR Valuation Adjustment Mechanism
争点词：效力 OR 有效 OR 无效 OR 违反公司资本维持 OR 违反公司法强制性规定
判决词：（案 OR 判决 OR 裁定）
监管词：（反馈 OR 问询 OR 处罚）
```

### 2.2 来源优先级

每个来源都对应一个**官方检索入口**，优先级由高到低：

| 优先级 | 来源 | 入口 | 案件类型 |
|-------|------|------|---------|
| P0 | 中国裁判文书网 | wenshu.court.gov.cn | 司法裁判（最高法 / 高院 / 中院 / 基层） |
| P0 | 证监会官网 | www.csrc.gov.cn → 行政处罚 | 证监会行政处罚决定书 |
| P0 | 上交所官网 | www.sse.com.cn → 纪律处分与监管措施 | 上交所纪律处分决定书 |
| P0 | 深交所官网 | www.szse.cn → 监管信息公开 | 深交所纪律处分决定书 |
| P0 | 北交所官网 | www.bse.cn → 监管信息公开 | 北交所纪律处分决定书 |
| P1 | 证监会并购重组委 | 证监会官网 → 并购重组委 | 重组委审核公告 |
| P1 | 上交所上市委 | 上交所官网 → 科创板股票审核 → 审核动态 | 科创板上市委审议公告 |
| P1 | 深交所上市委 | 深交所官网 → 创业板注册制审核 | 创业板上市委审议公告 |
| P1 | 北交所上市委 | 北交所官网 → 公开发行并上市审核 | 北交所上市委审议公告 |
| P2 | 最高人民法院公报 | court.gov.cn → 公报案例 | 公报案例 / 指导性案例 |
| P2 | 国家法律法规数据库 | flk.npc.gov.cn | 法律文本（辅助佐证） |
| P3 | 商业数据库 | 北大法宝 / 威科先行（用户若已订阅） | 汇总 + 关键词检索 |

**原则**：
- P0 级来源**必须尝试**。若某 P0 源未访问到，在输出的"3. 时效性与局限性"里明确列出。
- P1 级在主题涉及 IPO 审核或并购重组审议时**必须尝试**，其他情形选择性。
- P2 级作为补充，保障法源完整性。
- P3 级可信度高但依赖用户订阅，若用户未明示有订阅，不默认尝试。

### 2.3 筛选规则

按以下顺序筛选检索结果，选出"典型案例"：

1. **直接相关**：案件事实或争点与研究主题高度相似
2. **审级权重**：最高法 > 高院 > 中院 > 基层法院；监管案例中证监会 > 交易所
3. **时间权重**：**注册制改革（2019 年科创板设立）之后的案例**给更高权重；近 2 年案例默认优先
4. **结论清晰**：有明确裁判主文 / 处罚决定 / 审议意见 > 审理中 / 一般性发问
5. **代表性**：覆盖不同裁判口径的案例各保留 1–2 个（避免过度偏向一种观点）

**数量上限**：单次检索输出**不超过 10 个**典型案例。若发现的案例数超过 10 个，输出"代表性抽样"+ 附录列出其余案例的案号清单。

## Phase 3 — 执行检索

### 3.1 执行模式

**模式 A：WebSearch 可用（默认模式）**

1. 按 Phase 2 的关键词矩阵 × 来源优先级逐一发起搜索
2. 对每个结果评估相关度，保留 P0/P1 源的高相关结果
3. 对**关键案例**（用户主题高度相关的 1–3 个）使用 WebFetch 提取官网详情页全文
4. 从全文中抽取字段：案号/文书号、当事人、事实、争点、裁判理由、结论

**模式 B：仅用户上传文件**（无 WebSearch / WebFetch）

1. 请求用户上传可检索的文本（判决书 PDF、处罚决定 PDF、审议公告截图等）
2. 调用 `pdf` / `docx` 外部 skill 提取文本（若可用）
3. 按字段抽取规则处理
4. 全输出标注"来源：用户上传文件，未独立核对公开数据库"

**模式 C：仅模型记忆**（外部检索全部不可用且无上传文件）

1. 在输出顶部**置顶声明**：
   > ⚠️ 本次输出未经独立检索核对，所有案例均为**待核实**状态。律师**不得**直接作为意见书或
   > 尽调报告的依据，必须亲自到裁判文书网 / 证监会官网 / 交易所官网核对原文后方可引用。
2. 对每条案例标注可靠性等级 **D**
3. 在"4. 下一步建议"中明确列出用户应访问的核实路径
4. **绝不编造**虚假案号、虚假文书号——若记忆模糊，宁可写"类似争点的案例在最高法判决中多见，
   但 Claude 未能从记忆中给出可靠案号，请律师以关键词 'xxx' 在裁判文书网检索后引用具体案号"

### 3.2 链接与引用安全

- 输出中引用的所有 URL 必须是**官方/权威来源**（见 Phase 2 的来源表）。不得链接到未经验证的
  第三方博客、百度文库、个人网站等。
- 裁判文书网的具体判决 URL 常因改版而失效——**优先记录"案号"本身**，并给出裁判文书网首页
  URL 供用户自行检索。
- 若 WebFetch 被禁用或目标页 403，不强行尝试其他方式（如 curl、第三方缓存）。

## Phase 4 — 输出

**必须**按 [shared/templates/research-output-format.md](../../shared/templates/research-output-format.md)
的统一格式输出，不在本 skill 另定义格式。要点复述：

- 外层结构为 "0. 检索策略 / 1. 核心结论 / 2. 支撑材料 / 3. 时效性与局限性 / 4. 下一步建议" 四段
- 本 skill 的 **2. 支撑材料** 段下只用 **2.2 案例清单**（不用 2.1 法规清单，如需法规同时检索请调用
  `ecm-research-reg-search`）
- 每条案例按统一格式的"案例条目"子模板输出（[案件类型] 案号 / 事实 / 争点 / 裁判要旨 / 结论 / 启示 / 出处 / 可靠性）
- 每条结论方括号标注 [C-xx] 引用编号，便于读者追踪
- **时效性段落**必须列出：本次检索日期、截止日期、未覆盖的数据源

### 本 skill 的结论段建议写法

"1. 核心结论"应当是 **3–8 句话**，每句对应一个明确的法律/监管口径判断，例如：

> 1. 最高法 2019–2024 年的裁判实践确认，与股东对赌的条款一般有效 [C-01][C-02]；
> 2. 与公司对赌是否有效，以公司是否履行减资程序为关键审查点 [C-03]；
> 3. 证监会审核口径要求 IPO 申报前对对赌条款进行**无条件清理**，注册阶段不得附条件恢复 [C-05]；
> 4. ……

结论后必须对应具体案例编号。若某结论仅有 D 级（记忆）支撑，在结论本身加"（待核实）"。

## 配置项

用户可在调用时覆盖以下默认值：

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| `time_window` | 近 5 年 | 检索时间范围，如"近 3 年"、"2019 年以来"、"2010–2024" |
| `sources` | 全部 P0 | 检索来源清单，可缩减为单一来源 |
| `max_cases` | 10 | 输出典型案例的最大条数 |
| `reliability_floor` | A/B/C/D | 输出的最低可靠性等级，默认 D（保留所有但标注），可收紧为 "B+"（只要 A 和 B 级） |
| `output_language` | 中文 | 中文/英文切换，默认中文 |

## 被其他 skill 调用的接口

本 skill 常被以下 skill 通过 "先检索 xxx 案例再继续" 的方式调用：

- `ecm-dd-*` 系列：DD 过程中遇到争议条款、实控人认定争点、同业竞争认定争点等时
- `ecm-design-ipo-path`、`ecm-design-deal-structure` 等：方案设计时需要同类交易的监管先例
- `ecm-draft-opinion-letter`、`ecm-draft-disclosure-review`：意见书论证需要案例支撑

被调用时，本 skill 的输出应保持结构化，便于上游 skill 直接提取"2.2 案例清单"段注入其论证。

## 硬性输出要求

1. **遵守统一输出格式**：必须引用 `shared/templates/research-output-format.md`，不得自定义段落顺序
2. **可靠性等级必须标注**：每条案例必注 A/B/C/D
3. **出处必须可核对**：每条案例给出官方 URL 或数据库名 + 访问日期
4. **不得编造案号**：记忆模糊时明确标注"待核实"
5. **链接安全**：仅链接官方/权威来源，不链接第三方博客或不明来源
6. **不超过 10 条典型案例**：超出的以附录案号清单形式列出
7. **时效性必须明示**：最新案例检索截止日期、未覆盖的数据源均需写入"3. 时效性与局限性"
