# 股东大会见证意见内核审查 Skill

## 用途概述

本 skill 面向中国 A 股上市公司/新三板挂牌公司股东（大）会法律见证业务，
扮演"内核小组"角色，对项目组提交的《法律意见书》（见证意见）
进行形式 + 实质 + 常见错误三维度审查，产出一份带"内核"
作者修订痕迹和批注的 Word 文档。

本 skill 依据以下公开规则体系：
- 《公司法》（2024 年修订）
- 《上市公司股东会规则》（2025 年修订，原名《股东大会规则》）
- 《律师事务所从事证券法律业务管理办法》（中国证监会 2019 年第 223 号令）
- 《律师事务所从事证券法律业务执业规则》（中华全国律师协会）第 29–38 条、第 34 条
- 《公开发行证券公司信息披露的编报规则第 12 号——公开发行证券的法律意见书
  和律师工作报告》第 9 条、第 19–23 条
- A 股上市公司股东（大）会见证法律意见书行业通用起草指引模板

## 输出格式契约（三条硬性要求）

**这三条来自用户的明确要求，不得偏离。产出文档必须同时满足：**

### 1. 修订者 = "内核"

所有 `<w:ins>` / `<w:del>` 和 `<w:comment>` 的 `w:author` 属性
**必须**统一写为 `内核`（不是默认的 "Claude"、"Anthropic"
或签字律师姓名）。这覆盖 docx skill 默认使用 "Claude" 的行为。

- 插入 `comment.py` 时必须加 `--author "内核"`
- 直接编辑 document.xml 写 `<w:ins>` `<w:del>` 时必须用 `w:author="内核"`

### 2. 修订模式开启 + 最小显示改动原则

打开 Word 修订模式，让律师能直接看到删/增痕迹。
**关键原则**：只标记真正变动的字符，不要"整段删除 + 整段重写"。

正确示例（原句"我爱你" → 新句"我恨你"）：

```xml
<w:r><w:t>我</w:t></w:r>
<w:del w:author="内核" w:date="..."><w:r><w:delText>爱</w:delText></w:r></w:del>
<w:ins w:author="内核" w:date="..."><w:r><w:t>恨</w:t></w:r></w:ins>
<w:r><w:t>你</w:t></w:r>
```

错误示例（不要这样做）：

```xml
<w:del w:author="内核"><w:r><w:delText>我爱你</w:delText></w:r></w:del>
<w:ins w:author="内核"><w:r><w:t>我恨你</w:t></w:r></w:ins>
```

实务中高频应用场景：
- "2024 年" → "2025 年"：只删/增 "4" → "5"，不要整串改
- "第一次临时股东大会" → "第二次临时股东大会"：只删/增 "一" → "二"
- "9:30" → "14:30"：只删/增 "9" → "14"
- 删除一句不适用的模板话（如网络投票描述）：只删除该句，保留其他

### 3. 解释文字只进批注，不进正文

**正文里禁止出现任何解释性/说明性文字**。所有需要说明的内容
（疑问、要求落实底稿、提醒项目组核查、法规依据、建议）
全部通过 `<w:comment>` 批注呈现。

- ✅ 批注里可以写："内核提示：该议案为特别决议事项，请根据见证业务
      起草指引模板补充'本项议案为股东（大）会特别决议事项，已经出席本次
      股东（大）会的股东及股东代理人所持表决权的三分之二以上同意通过'"
- ❌ 不要直接把这句话用 `<w:ins>` 插进正文中

批注可用分类前缀（便于项目组分诊，但不是强制）：
- **【必改】** 硬性错误（事实错、法规错、模板残留、基础信息错）
- **【核实】** 需与公司/项目组确认再定的问题
- **【建议】** 措辞优化、行文一致性、体例统一
- **【底稿】** 需落实底稿或更新查验计划的事项

## 工作流（五步）

### Step 1 — 定位和预读入文件

```bash
ls /mnt/user-data/uploads/
```

找到待审文件（通常是 `.docx`；如是 `.doc` 先用
`scripts/office/soffice.py --headless --convert-to docx` 转换）。

先用 `extract-text` 把全文读一遍（若有已存在的修订痕迹，用
`pandoc --track-changes=all` 查看）：

```bash
extract-text /mnt/user-data/uploads/xxx.docx > /tmp/review-input.md
```

通读全文形成整体理解：这是哪家公司？哪次会议（年度/临时、届次）？
审议了什么议案？有无特别决议/关联交易/累积投票/独立董事选举？

**并在这一步同时提取三个信息用于 Step 1.5**：
- **股票代码**（6 位数字，通常在开头"致 ×××× 公司"处或签字页公司全称附近能找到）
- **板块**（沪市 6 开头 / 深市 0/3 开头 / 北交所 4/8 开头 → `市场=沪深京`；
  新三板挂牌公司 → `市场=三板`）
- **股东大会召开日期**（YYYYMMDD 格式）

如果股票代码从见证意见里识别不出来（比如仅提及公司简称未给代码），用 web_search
以"公司全称 股票代码"检索一下即可；再不行就直接进入 Step 1.5 的 Level 2（提示用户
补充信息或上传参考文件）。

### Step 1.5 — 拉取并解析参考文件（三级降级）

这一步是为了拿到两份事实核对的"底稿"：**股东大会通知**和**发出这次股东
大会通知的董事会决议公告**。它们是交叉比对的依据——见证意见里写的日期/
届次/议案等字段，都要和这两份文件对照。

**为什么这一步很关键**：行业常见错误模式里最高频的错误是"日期/时间/届次与通知
不一致"（见 `common-errors.md` A 类），这类事实错**没有参考文件是审不出来的**，
内核批注的含金量也会大打折扣。

按以下三级降级执行：

#### Level 1 — 自动拉取

运行脚本：

```bash
python /path/to/skill/scripts/fetch_cninfo_announcements.py \
    --stock-code <股票代码> \
    --meeting-date <YYYYMMDD> \
    --market <沪深京|三板> \
    --output-dir /home/claude/refs
```

（脚本第一次运行前确保依赖齐全：`pip install requests pypdf --break-system-packages`）

脚本返回 JSON 到 stdout，关键字段：
- `shareholders_notice`：股东大会通知的元数据 + PDF 路径 + 文本路径（`text_path`）
- `board_resolution`：董事会决议公告的元数据 + PDF 路径 + 文本路径
- `shareholders_notice_candidates` / `board_resolution_candidates`：所有候选
  （当脚本自动挑错时可手动从候选里选）
- `errors`：遇到的问题列表

退出码：
- **0** — 两份都拿到 → 读取两份 .txt 作为后续交叉比对的依据，进入 Step 2
- **1** — 只拿到一份 → 另一份走 Level 2（提示用户上传缺的那份）
- **2** — 一份都没拿到 → 走 Level 2（提示用户上传两份）

**自动挑选的校验**：脚本会按启发式规则自动挑"最像的那份"，但可能挑错
（例如同一天披露多份董事会决议公告时）。拿到结果后对照见证意见里提到的
董事会届次、股东大会届次，**必要时从候选列表里手动确认**——如果自动挑的
不对，从候选里找正确的 `pdf_url` 重新下载（或提示用户）。

#### Level 2 — 提示用户上传

如果 Level 1 因任何原因未能拿到需要的文件（网络失败、公告未披露、股票代码
识别不出、退市等），向用户明确发出上传请求：

> 【内核提示】自动从巨潮资讯网拉取参考文件未成功（原因：……）。为了做事实
> 性交叉比对，请上传以下文件：
> 
> - [公司简称] [届次] **股东大会通知**（PDF 或 docx）
> - 对应的**董事会决议公告**（即决定召开本次股东大会的那次董事会决议公告）
> 
> 上传后我会继续审查。如果暂时拿不到这两份文件，请回复"跳过比对"，
> 我将继续审查但不做交叉核对。

等用户回复：
- 用户上传了文件 → 读取文件内容（PDF 用 pypdf，docx 用 `extract-text`），
  作为后续交叉比对的依据，进入 Step 2
- 用户只上传了其中一份 → 有哪份用哪份，另一份标为缺失
- 用户回复"跳过比对"或类似表述 → 进入 Level 3

#### Level 3 — 跳过交叉比对，显式告知用户

如果用户确认无法提供参考文件，**在最终输出文档的开头插入一条整体性
批注**（位置：文档标题或收件人段首），明确告知：

> 【内核综合意见】本次内核审查**未进行与股东大会通知、董事会决议公告的
> 交叉核对**（因参考文件未能获取/提供）。因此：
> 
> - 基础信息类字段（召开日期/时间/地点/届次/股权登记日/议案清单/
>   董事会届次等）**未经外部文件核对**，请项目组自行对照《股东大会通知》
>   和相应《董事会决议公告》核查；
> - 本次批注仅基于见证意见文本自身的形式审查和实质审查完成。
> 
> 如需完整内核，请补充上述参考文件后重新提交。

然后只做原有的三维度审查（形式 + 实质 + 常见错误库扫描），不写任何
依赖参考文件证据的批注（即 `cross-check-matrix.md` 里"带参考文件证据版"
的话术都不用）。

#### Level 1 成功后的后续使用

读取 `shareholders_notice.text_path` 和 `board_resolution.text_path` 两个
.txt 文件的内容（`extract-text` 或直接读），作为 Step 2 审查时的"事实
依据"。交叉核对的完整矩阵参见 `references/cross-check-matrix.md`。

---

### Step 2 — 三维度审查（同步记录 issues）

建一个 issue 清单，每条记录：
- **位置**（章节/段落/句子）
- **原文摘录**（几个字即可，用于后面定位）
- **问题类型**（形式 / 实质 / 常见错误）
- **处理方式**（tracked change 改正文 / comment 加批注 / 两者并用）
- **修改后正文 or 批注文案**

**参考清单依次过一遍**（顺序：先做交叉比对，再做清单审查）：

0. **交叉比对审查**（仅 Step 1.5 成功获得参考文件时执行）— 阅读
   `references/cross-check-matrix.md`，按 12 项核心字段 + 推导性检查，把见证
   意见、股东大会通知、董事会决议公告三者对照，标出任何事实性不一致。此步
   **优先级最高**——事实错是行业常见错误模式里的最高频类别。

1. **形式要件审查** — 阅读 `references/form-requirements.md`，
   按 14 项清单逐条勾对。

2. **实质审查** — 阅读 `references/substantive-checklist.md`，
   按实质审查清单 8 类审查项逐项过。

3. **常见错误扫描** — 阅读 `references/common-errors.md`，
   按行业常见错误模式筛查（日期/届次/特别决议标记/累积投票表述
   /《股东会规则》更新/模板残留等）。

**跨字段一致性检查**（很关键，常见扣分点）：
- 召开日期/时间：见证意见内 vs 股东会通知 vs 决议公告 三者必须一致
- 届次（第几届第几次董事会、第几次临时股东会）：全文统一
- 公司简称：全文首现定义 → 后续使用 → 决议页 → 签字页，必须一致
- 年份：2025 年的会议不能出现"2024 年"字样（高频错误）
- 议案列表：通知里列的议案 vs 意见书第三章列的 vs 决议内容列的，
  逐项对应无遗漏

### Step 3 — 准备修订工作目录

```bash
cd /home/claude
cp /mnt/user-data/uploads/xxx.docx ./input.docx
python /mnt/skills/public/docx/scripts/office/unpack.py input.docx unpacked/
```

`unpack.py` 会自动美化 XML、合并相邻 run、转换智能引号。
解包后主要编辑 `unpacked/word/document.xml`。

### Step 4 — 写入修订痕迹和批注

**(A) 修订痕迹（tracked changes）——用 Edit 工具直接改 document.xml**

严格遵循"最小显示改动"原则。典型模式：

**替换几个字**（如"2024" → "2025"）：

```xml
<w:r><w:t>截至</w:t></w:r>
<w:del w:id="1" w:author="内核" w:date="2026-04-23T00:00:00Z">
  <w:r><w:delText>2024</w:delText></w:r>
</w:del>
<w:ins w:id="2" w:author="内核" w:date="2026-04-23T00:00:00Z">
  <w:r><w:t>2025</w:t></w:r>
</w:ins>
<w:r><w:t>年</w:t></w:r>
```

**删除一句**（如挂牌公司残留的《股东大会规则》引用）：把该句
所在的 `<w:r>` 替换成 `<w:del>` 包裹的 `<w:delText>`。如果删的
是整段，还要在 `<w:pPr><w:rPr>` 里加 `<w:del/>` 标记段落标记被删，
避免留下空段。

**插入新内容**（极少用——大部分新增建议都应该放批注里）：
仅在缺漏必要模板文字且没有替代表达时才用 `<w:ins>` 直接补进正文。
情况不确定（比如不知道确切票数）时，一律放批注让项目组自行核实。

**关键 XML 细节**：
- 每条修订用唯一的 `w:id`（1, 2, 3... 自增，与 comment 的 id 独立）
- 日期用 `2026-04-23T00:00:00Z` 格式（今天是 2026-04-23）
- `<w:del>` 内部用 `<w:delText>` 不是 `<w:t>`
- 替换整个 `<w:r>` 块，不要把 `<w:ins>` `<w:del>` 塞进 run 内部
- 原 `<w:r>` 若有 `<w:rPr>` 格式设定（加粗、字体、字号），要复制到
  新的 del/ins 的 `<w:r>` 里，保持格式一致

**(B) 批注（comments）——用 comment.py**

对每条批注：

```bash
python /mnt/skills/public/docx/scripts/comment.py unpacked/ <N> \
  "批注文本（XML 实体已转义的）" \
  --author "内核"
```

其中 `<N>` 是批注 ID（0, 1, 2... 自增）。批注文本如含 `&`
要写成 `&amp;`，含 `'` 用 `&#x2019;`，含 `"` 用 `&#x201C;` / `&#x201D;`。

然后在 document.xml 相应位置插入三个标记（commentRangeStart /
commentRangeEnd / commentReference），这三个标记都是 `<w:p>` 的
直接子节点，**不能**套在 `<w:r>` 里。范例：

```xml
<w:commentRangeStart w:id="0"/>
<w:r><w:t>被批注的原文内容</w:t></w:r>
<w:commentRangeEnd w:id="0"/>
<w:r><w:rPr><w:rStyle w:val="CommentReference"/></w:rPr>
  <w:commentReference w:id="0"/>
</w:r>
```

可以在同一段文字上同时有 tracked change + comment：
commentRangeStart → `<w:del>` + `<w:ins>` → commentRangeEnd →
commentReference。

**(C) 批注撰写风格** — 参考 `references/comment-templates.md`，
使用行业通用的内核批注标准话术。核心原则：
- 明确指向问题，给出修改方向或落实要求
- 必要时引用法规依据（《公司法》《股东会规则》《管理办法》具体条文）
- 对项目组的行动项用动词开头："请补充……"、"请核查……"、
  "请落实底稿……"

### Step 5 — 打包、输出、呈递

```bash
python /mnt/skills/public/docx/scripts/office/pack.py \
  unpacked/ /mnt/user-data/outputs/reviewed.docx \
  --original input.docx
```

`pack.py` 会自动校验 + 小修复（如 durableId 越界、`xml:space`
缺失）。如果校验报错是结构性错误，需要按报错定位到 XML 修复后
重新 pack。

输出文件名建议体现"内核后"语义，例如
`[公司简称]_[届次]_见证意见_内核后_[日期].docx`。

最后用 `present_files` 工具呈递给用户：

```python
present_files(filepaths=["/mnt/user-data/outputs/reviewed.docx"])
```

附一段简短的总结性回复：用了几条 tracked change、几条 comment，
发现的主要问题类别是什么（如"3 条日期/届次事实错、2 条特别决议
表述缺失、4 条批注提醒项目组落实底稿"），**不要**把全部修改内容
重复罗列——律师会在 Word 里直接看。

## 边界与谨慎处理

**本 skill 不做的事情**：
- 不主动替代律师做法律判断——拿不准的事项一律用【核实】类批注
  提示项目组进一步核查
- 不修改项目组已写明的结论性意见（"合法合规"/"真实有效"等），
  除非存在明显的事实冲突；有疑虑的一律批注提示
- 不访问公开信息做事实核查——无法查证的时间/届次/公告编号等
  一律用【核实】批注要求项目组自行核对，而不是猜测改动

**遇到原文与本 skill 规则不匹配的"可能疏漏"**（例如文件里根本没有
声明事项段）：以【必改】批注提示补充，不自行在正文里补写整段声明。
