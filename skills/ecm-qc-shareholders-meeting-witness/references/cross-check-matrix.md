# 交叉比对矩阵（Cross-Check Matrix）

本文件规定**在已获得"股东大会通知"和"董事会决议公告"两份参考文件的情况下**，
内核审查应逐字段进行的一致性核对。这是本 skill 的核心增量价值——**没有参考
文件时做的只是语法和体例审查，有了参考文件才能做事实性的交叉核对**，而事实错
恰恰是行业常见错误模式里最高频的类别（A 类常见错误中"日期/时间/届次"三连错占了
过半比例）。

---

## 工作原则

### 原则 1：三源对照，逐字精确比对

凡参考文件中存在的字段，**都要把见证意见里的写法、股东会通知里的写法、董事会
决议公告里的写法三者摆出来对照**。

**逐字精确**——多一个字、少一个字、空格 / 标点 / 全半角差异都要核对。

**逐字精确适用范围（必须全覆盖）**：

| 字段 | 高频漏点 |
|---|---|
| 召开日期 / 时间 | 上下午、时点 |
| 召开地点 | 楼层 / 会议室号、A3 会议室 / 3004 等门牌 |
| **议案名称** | **简称 vs 全称、标点、文件名书名号** |
| 股权登记日 | — |
| 召集人 | 公司董事会 vs 公司董事 |
| 网络投票时间 | 起止时点（H 股可能纯现场无网络）|
| 主持人 | 姓名、职务 |
| 股东会届次 | "2025 年年度" vs "2025 年度" |

**典型实战漏识别**：

> **场景 A — 地点多字**（成都纵横自动化）：
> - 通知地点："成都市高新区天府五街 200 号菁蓉汇 3A11 楼"
> - 见证意见："……3A11 楼 **A3 会议室**" → 多了"A3 会议室"

> **场景 B — 地点少字**（中金公司三会）：
> - 通知地点："国寿金融中心 30 层 **3004 及 3005 会议室**"
> - 见证意见：仅"国寿金融中心 30 层" → 漏会议室号

> **场景 C — 议案名称简称 vs 全称**（中金公司三会）：
> - 通知简称：《关于本次交易符合《上市公司监管指引第 9 号》第四条规定的议案》
> - 见证意见全称：《关于本次交易符合《上市公司监管指引第 9 号——上市公司
>   筹划和实施重大资产重组的监管要求》第四条规定的议案》
> - 通知用简称的（即使通知备注"议案名称详见会议资料"），**见证意见也应按通知
>   统一为简称**，不能擅自换成全称——【必改】

**议案名称比对的硬规则**：

- 不允许"通知简称 → 见证意见全称"的自动替换
- 不允许标点 / 书名号位置差异
- 不允许字号简称差异（"中金公司"vs"中国国际金融股份有限公司"——前后必须统一）
- 子议案编号 ".00"vs"" 必须与通知一致

### 原则 2：三方任一不一致都要批注（不只是"见证意见错"）

实战中常出现"公司自己在董事会决议公告或股东会通知里写错"——此时见证意见
反而是正确的。这种情况**不要改见证意见**，但**必须批注提示**：

> **德联集团 2025 年度股东会**（实战漏识别）：
> - 见证意见：2026 年 5 月 25 日（与实际召开日、与通知一致）✓
> - 董事会决议公告：2026 年 5 月 **14** 日 ❌（公司自己写错了）
> → 见证意见不改；写【核实】批注："董事会决议公告中股东会召开日期 5/14 与
>   本意见书及通知中 5/25 不一致，请核实是否有补充公告或决议公告系笔误。"

**判定路径**：

```
比对三个字段值 → 三者完全一致 → 不批
                  → 任意一方与其他不一致 → 必批，按下表处理
```

| 不一致情形 | 处理 |
|---|---|
| 见证意见 ≠ 通知 = 决议公告 | tracked change 改见证意见 + 【必改】 |
| 见证意见 = 通知 ≠ 决议公告 | 不改见证意见 + 【核实】批注（决议公告可能写错） |
| 见证意见 = 决议公告 ≠ 通知 | 不改见证意见 + 【核实】批注（通知可能写错） |
| 三者两两都不同 | 不改见证意见 + 【必改】批注，要求项目组与公司确认正确值 |

### 原则 3：以"最源头"为准（仅用于决定 tracked change 改正方向）

- 董事会是否召开 / 召开日期 / 届次 → 以**董事会决议公告**为准
- 股东会召开日期 / 时间 / 地点 / 议案 → 以**股东会通知**为准
- 但**最源头本身可能写错**——见原则 2

### 原则 4：处理方式

- 事实错（可直接从参考文件取到正确值，且参考文件之间一致）→ tracked change
  + 批注注明来源
- 三方矛盾 → 不改见证意见，只【核实】批注
- 参考文件里没有的字段（如关联股东身份隐性核查）→ 【核实】/【底稿】批注

批注引用参考文件时用明确表述，例如：

> 【必改】经核对公司于 YYYY-MM-DD 披露的《XX 股东会通知》（公告编号
> XXXXXXX），本次股东会召开地点应为 [完整地点]，见证意见此处多了"A3 会议室"
> 等字样，请订正。

或：

> 【核实】《第 X 届董事会第 Y 次会议决议公告》中提及股东会召开日期为 YYYY-MM-DD，
> 与本意见书及《股东会通知》载明的 YYYY-MM-DD 不一致。请核实是否系决议公告
> 笔误，或公司另发了补充公告 / 重新发了董事会决议。

## 核心交叉核对字段（12 项）

下表中"✓"表示该字段应在该文件里存在并核对；"—"表示通常不含。
"权威性"表示不一致时以哪个文件为准。

| 字段 | 见证意见 | 股东大会通知 | 董事会决议公告 | 权威性 | 处理 |
|---|:-:|:-:|:-:|---|---|
| 股东大会**召开日期** | ✓ | ✓ | ✓（决定召开日期） | 股东大会通知 | tracked change + 【必改】 |
| 股东大会**召开时间**（含上/下午、具体时点） | ✓ | ✓ | — | 股东大会通知 | tracked change + 【必改】 |
| 股东大会**召开地点** | ✓ | ✓ | — | 股东大会通知 | tracked change + 【必改】 |
| 股东大会**届次**（"2025年年度"/"2025年第X次临时"） | ✓ | ✓ | ✓ | 股东大会通知 | tracked change + 【必改】 |
| **股权登记日** | ✓ | ✓ | — | 股东大会通知 | tracked change + 【必改】 |
| **网络投票安排**（平台、开通/截止时间） | ✓ | ✓ | — | 股东大会通知 | tracked change + 【必改】 |
| **审议议案清单** | ✓ | ✓ | ✓（董事会提交部分） | 股东大会通知（总清单）+ 董事会决议（来源） | 缺失/多余：【必改】 |
| **特别决议议案**的标注 | ✓ | ✓ | ✓ | 股东大会通知 | 标注缺失：【必改】 |
| **关联股东回避**名单 | ✓ | ✓ | — | 股东大会通知 | 缺失/不一致：【必改】 |
| **中小投资者单独计票**议案标注 | ✓ | ✓ | — | 股东大会通知 | 缺失：【必改】 |
| **董事会届次**（"第 X 届董事会第 Y 次会议"） | ✓ | ✓（通知引用） | ✓ | 董事会决议公告 | tracked change + 【必改】 |
| **董事会召开日期** | ✓ | ✓（通知引用） | ✓ | 董事会决议公告 | tracked change + 【必改】 |

**加分项**（不在核心 12 项，但有参考文件就能顺手做的）：

| 字段 | 见证意见 | 股东大会通知 | 董事会决议公告 | 说明 |
|---|:-:|:-:|:-:|---|
| 公告**披露日期**（董事会决议 + 股东大会通知） | ✓ | ✓（自身） | ✓（自身） | 用于计算是否满足法定通知期（年度 ≥20 日、临时 ≥15 日、新三板临时 ≥10 日） |
| 公司**全称**与**简称** | ✓ | ✓ | ✓ | 跨文档简称一致性检查 |

## 推导性检查（基于参考文件可算出来的）

这几项不是简单字段对照，而是需要基于参考文件里的字段做计算：

### 1. 法定通知期是否满足

- **年度股东大会**：股东大会通知披露日 → 召开日之间 **≥ 20 个自然日**
- **临时股东大会**（沪深）：**≥ 15 个自然日**
- **新三板临时股东会**：**≥ 10 个自然日**
- 计算方式：股东大会通知的"公告时间"（从 cninfo `announcementTime` 字段得到）
  → 见证意见/通知中载明的"召开日期"

不足的处理：
> 【必改】经核对，《股东大会通知》披露于 YYYY-MM-DD，本次股东大会召开日为
> YYYY-MM-DD，通知期为 N 日，**不足**《公司法》第 XX 条及公司章程第 XX 条
> 规定的 M 日法定通知期，请项目组核查。

### 2. 董事会召集 → 股东大会通知 → 股东大会召开 的时序逻辑

合规时序应是：董事会决议日 ≤ 股东大会通知披露日 ≤ 股东大会召开日。
如果出现倒挂（比如见证意见写的董事会日期 > 通知披露日期），几乎一定是事实错。

### 3. 议案清单完整性矩阵

把三份文件里的议案逐条摆出来画个对照表：

| 议案编号 | 股东大会通知中 | 董事会决议公告中（提交审议） | 见证意见中审议章节 | 见证意见中决议章节 |
|---|---|---|---|---|

三列都有 × 见证意见两列都有 = PASS。任何一列缺失都要批注。
**特别注意**：有"总议案"和"子议案"结构的（如议案 1 下设 1.1、1.2、1.3），
见证意见须对总议案和每个子议案分别写审议结果。

### 4. 特别决议 / 累积投票 / 中小投资者单独计票的标注一致性

> **强校验逻辑**：本节给出与通知交叉比对的判定规则；议案性质识别关键词、
> 法规依据（《股东会规则》2025 新版第 32/33/46 条等）、完整批注模板请阅
> `voting-rules.md`。

#### 4.1 三步法

```
(a) 对每条议案打三个标签（特别决议 / 累积投票 / 单独计票）
    依据：voting-rules.md §5 关键词识别矩阵
(b) 从股东会通知中摘出三类议案的标注原文
(c) 用矩阵 X / Y / Z（见 voting-rules.md §6）做强校验
```

**写批注硬规则**：发现问题才批注；通知未标但议案性质应当属于的，用【核实】
不在正文改判；**禁止"已核查无问题"类的总结性批注**。

#### 4.2 总议案 vs 子议案（特别说明）

默认情况下，议案被拆成子议案后，见证意见**只列示子议案表决结果，不再列示
总议案**——这是行业通行做法，不需要批注。

只有当**通知的提案编码表把总议案单独编码并标"作为投票对象的子议案数（N）"**
时（深交所典型做法），才可能存在总议案独立统计数据。此时若见证意见遗漏总
议案表决结果，写【核实】让项目组核查深圳证券信息有限公司是否提供了总议案
的独立票数；其余情形不批。

#### 4.3 矩阵填写

完成 Step 1.5 拿到通知后，在 issue 清单内部建立三张表（不写入批注）：

**矩阵 X — 特别决议**

| 议案 | 通知标 | 性质应是 | 见证写 2/3 | 比例 ≥ 2/3 |
|---|:-:|:-:|:-:|:-:|

**矩阵 Y — 累积投票**

| 议案 | 通知标 | 选举且触发强制条件 | 见证说明累积 | 仅同意票 | 当选结论 |
|---|:-:|:-:|:-:|:-:|:-:|

**矩阵 Z — 中小投资者单独计票**

| 议案 | 通知标 | 性质应是 | 见证列单独计票 |
|---|:-:|:-:|:-:|

具体判定规则（哪种组合 → 哪种【必改】/【核实】）和批注模板见 `voting-rules.md`
§6/§8。

## 输出到批注的标准话术（带参考文件证据版）

相比 `comment-templates.md` 里的通用话术，有了参考文件后批注应当**引用
参考文件的具体字段作为证据**，这样项目组看到批注能直接复核，不用再翻原始公告。

### 日期/时间/届次类

> 【必改】经核对《XX 股东大会通知》（披露于 YYYY-MM-DD，公告编号 XXXXX），
> 本次股东（大）会召开日期/时间/地点/届次应为「XX」，见证意见此处表述为「YY」
> 与通知不符，请订正。

### 董事会届次/日期类

> 【必改】经核对《第 X 届董事会第 Y 次会议决议公告》（披露于 YYYY-MM-DD），
> 决定召开本次股东（大）会的董事会为"第 X 届第 Y 次会议"，召开日期 YYYY-MM-DD。
> 见证意见此处表述为「...」，请核对订正。

### 议案缺失类

> 【必改】经核对《XX 股东大会通知》，本次股东（大）会审议议案包括议案 X
> "XXX"，但见证意见中未见对该议案的审议与表决段落，请补充。

### 特别决议标注类

> 【必改】《XX 股东大会通知》明确议案 X "XXX"为股东（大）会特别决议事项，
> 见证意见未标注，请补充表述："本项议案为股东（大）会特别决议事项，已经
> 出席本次股东（大）会的股东及股东代理人所持表决权的三分之二以上同意通过。"

### 中小投资者单独计票类

> 【必改】《XX 股东大会通知》明确议案 X 为中小投资者单独计票议案，见证意见
> 未列明该议案的中小投资者表决结果（同意/反对/弃权票数及比例），请补充。

### 关联回避类

> 【必改】《XX 股东大会通知》明确关联股东 [A、B、C] 应回避议案 X 的表决。
> 见证意见回避名单为 [X、Y]，与通知不一致，请核对订正。

### 法定通知期不足类

> 【必改】《XX 股东大会通知》披露于 YYYY-MM-DD，本次会议召开日为 YYYY-MM-DD，
> 间隔 N 日，不足《公司法》关于[年度/临时]股东大会提前 M 日通知的规定，请核查
> 补救措施（如补充披露、推迟召开等）。

## 特殊场景

### 场景 A：找到多份董事会决议公告候选

同一公司往往一年开多次董事会。`fetch_cninfo_announcements.py` 脚本默认挑选
"公告时间最接近股东大会召开日、且在其之前"的那份。但如果候选列表中有多份
都很接近（比如同一天披露了两份董事会决议公告），Claude 应当：

1. 查看见证意见里引用的董事会届次（如"第八届董事会第十五次会议"）
2. 在候选列表（`board_resolution_candidates`）中找标题精确匹配该届次的那份
3. 如果脚本自动挑的不对，用 `board_resolution_candidates` 里该项的 `pdf_url`
   手动下载（或改用该 announcement_id 再跑一次脚本）

### 场景 B：找到的股东大会通知不止一份

常见于有"补充通知"或"更正通知"的情况。脚本默认排除标题含"补充/更正"的，
只挑主通知。如果见证意见明确基于"补充通知"（即通知内容经修订后召开），
Claude 应当同时参考主通知和补充通知。

### 场景 C：公司更改披露板块或新老代码转换

如北交所新老代码对照情形，用老代码搜不到时用新代码再试。
脚本的 `--market` 参数允许在"沪深京"和"三板"之间切换。

### 场景 D：挂牌公司（新三板）

- `--market` 传 `三板`
- 审查时移除《上市公司股东会规则》的适用性判断（这是挂牌公司的高频扣分点）
- 通知期按新三板规则：年度 20 日、临时 10 日

## 如果没有参考文件（降级路径）

SKILL.md 的 Step 1.5 规定三级降级：自动拉取 → 用户上传 → 跳过交叉比对。
如果最终没有参考文件，本矩阵中**所有依赖参考文件证据的批注都不要写**——
只做 `form-requirements.md` / `substantive-checklist.md` / `common-errors.md`
/ `voting-rules.md` 四个清单规定的审查（注意：**`voting-rules.md` 中基于
议案性质本身的识别仍可做**——三种特殊表决方式中相当一部分错误是见证意见
自身内部不一致，与是否有通知无关，例如累积投票却列出反对/弃权票、特别决议
表述与议案标题不匹配、单独计票口径混算等），并在输出文档开头加一条"整体性
批注"说明未做交叉比对，详见 SKILL.md Step 1.5。
