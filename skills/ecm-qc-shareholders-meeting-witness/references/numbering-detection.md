# Word 自动编号 vs 手动文本编号的识别（Numbering Detection）

## 为什么单独成篇

项目组常用 Word 自动编号（`<w:numPr>` + `numbering.xml`）和**手动文本编号**
（直接在 `<w:t>` 里写"一、""1.""（二）"）混搭。如果内核审查只看
`<w:t>` 文本而不解析 `<w:numPr>`，会出现典型的**编号误判**：

> **典型场景**（来自申通 2025 年度股东会见证意见）：
>
> - P42（`numId=22, ilvl=0, lvlText="（%1）"`）→ Word 渲染为 **"（一）本次
>   股东会的表决程序"**
> - P47（**无** `numPr`，文本直接写"（二）本次股东会的表决结果"）
>
> 只看文本会误以为"P47 是（二），但前面 P42 没有（一）"——其实 P42 是自动
> 编号生成的"（一）"，视觉上完全连贯，**不该批注**。

类似误判：

| 项目组写法 | `<w:t>` 看到的 | Word 实际渲染 | 误判 |
|---|---|---|---|
| 自动编号 "1." → 手动 "2." | "" → "2." | "1." → "2." | "前面没有 1，2 错了" |
| 自动 "一、" → 手动 "二、" | "" → "二、" | "一、" → "二、" | "二、 前面缺 一、" |
| 手动 "（一）" → 自动 "（二）" | "（一）" → "" | "（一）" → "（二）" | "（一）后面缺（二）" |

---

## 硬规则

**在写"编号不一致 / 编号错 / 编号缺失"类批注之前，必须先用脚本核对实际渲染
的编号**。具体操作：

```bash
# 一次性看整个文档每段的渲染编号
python3 scripts/render_paragraphs.py /path/to/file.docx

# 单独看某段的详情（确认它到底是自动还是手动）
python3 scripts/render_paragraphs.py /path/to/file.docx --para 47

# 看 numbering.xml 中所有 numId 的格式定义
python3 scripts/render_paragraphs.py /path/to/file.docx --numbering-summary
```

输出示例（每段一行）：

```
P 41 [三、] (numId=18 L0)   本次股东会的表决程序、表决结果
P 42 [（一）] (numId=22 L0)  本次股东会的表决程序          ← 自动编号
P 43 [1.] (numId=23 L0)     本次股东会审议的议案……
...
P 47                        （二）本次股东会的表决结果      ← 手动文本（无 numId）
```

- **方括号 `[（一）]`**：表示该段有自动编号，渲染后会显示这个值
- **括号 `(numId=... L0)`**：编号定义来源
- **没有方括号**：纯手动文本，`<w:t>` 里有什么就显示什么

## 决策逻辑

### 1. 渲染后连贯 → 不批注

例如 P42 渲染"（一）"、P47 文本"（二）"——视觉连贯。**不要批注**。即便源
代码是混搭，也不是律师层面的内核审查范围。如果想提示项目组体例统一，最多
写【建议】级别的批注。

### 2. 渲染后有错位 / 跳号 / 重复 → 批注

例如：

```
P 30 [1.] (numId=20 L0)   ……
P 31 [2.] (numId=20 L0)   ……
P 32                      4. ……         ← 文本写 "4."，渲染应该是 "3."
```

→ 【必改】P32 编号"4."应为"3."（与前段自动编号 P31"2."衔接）。

### 3. 渲染后视觉一致但源码混搭 → 不必批注；如批仅【建议】

像申通 P42/P47 的情况：

- **默认**：不批注（视觉上一致就是合规的）
- **如果项目组提了体例规范**：【建议】本节"（一）（二）"编号混用 Word 自动
  编号和手动文本，建议全部统一为自动编号或全部统一为手动文本，避免后续插入
  段落导致编号错位。

### 4. 跨章节编号格式不一致 → 看具体情形

例如：

```
P 21 [一、]    (numId=18 L0)  本次股东会的召集、召开程序
P 31 [二、]    (numId=18 L0)  出席本次股东会会议人员资格与召集人资格
P 41 [三、]    (numId=18 L0)  本次股东会的表决程序、表决结果
P104 [四、]    (numId=18 L0)  结论意见
```

→ 一致，不批。但如果出现：

```
P 41 [三、]    (numId=18 L0)  本次股东会的表决程序、表决结果
P 70           4. 结论意见                       ← 跳到 numId=24 体系或手动写错
```

→ 【必改】"结论意见"应使用与前面一致的"四、"编号格式。

## 在 issue 清单中标记

建议在 Step 2 整理 issue 时，**对任何涉及编号的疑问**都在条目里标记
"⚠ 需核对自动编号"，然后跑一遍 `render_paragraphs.py`，确认后再决定是否
保留为批注。

```
Issue 草稿：P47 编号"（二）"前面缺"（一）"  ⚠ 需核对自动编号
  → 跑 render_paragraphs.py --para 42 → 显示 P42 自动编号"（一）"
  → 删除本条 issue
```
