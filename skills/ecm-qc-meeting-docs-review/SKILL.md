# 会议文件内核审查 Skill

## 定位与边界

本 skill **负责**：
- 对项目组起草的会议文件成套（通知 / 议案 / 决议 / 记录 / 签到表 / 授权委托书 / 表决统计表 / 代理投票计票细则）进行**内核独立审查**
- 按三级工作流（跨文件一致性 + 形式 + 实质 + 常见错误扫描）产出带 tracked changes + comments 的 Word 文件（或批量时每份分别出）
- **参考坐标**：
  - [`shared/templates/meeting-docs-format.md`](../../shared/templates/meeting-docs-format.md) —— 会议文件起草规范（通知 / 议案 / 决议 / 记录 / 签到表结构 + 通知期限硬校验表 + 表决基数计算 + 特别决议 / 关联回避 / 中小投资者识别）
  - 同项目的 **公司章程**（如可获取；决定通知期限 / 表决比例 / 召开方式等"章程约定优先"的事项）
  - 同项目的 **DD-Memo-公司章程与治理**（`02-10-公司治理/`）和 **DD-Memo-批准和授权**（`02-01-批准和授权/`）
  - 同项目其他会议文件（互为参考源；通知 / 议案 / 决议 / 记录的议案清单应一致）

本 skill **不负责**：
- 起草会议文件 → `ecm-draft-meeting-docs`
- 股东大会**见证业务**（对见证法律意见书本身做内核）→ `ecm-qc-shareholders-meeting-witness`
- 律师工作报告 / 法律意见书审查 → `ecm-qc-work-report-review` / `ecm-qc-opinion-letter-review`
- 信披文件审查 → `ecm-qc-disclosure-review`
- 会议决议的内容合法性终局判断（如某项重大资产重组方案是否合法合规；由签字律师 + 独立财务顾问综合判断）
- Word 最终排版 → `ecm-draft-format-adjust`

本 skill 的骨架抽象自 [`shared/templates/qc-skill-template.md`](../../shared/templates/qc-skill-template.md)。

## 与 ecm-draft-meeting-docs / ecm-qc-shareholders-meeting-witness 的边界

### 与 ecm-draft-meeting-docs（起草）

| 维度 | ecm-draft-meeting-docs（起草） | ecm-qc-meeting-docs-review（内核审查 ← 本 skill） |
|------|---------------------------|----------------------------------------|
| 使用者 | 项目组起草人 | 内核 / QC 团队 |
| 视角 | "按模板批量生成一套会议文件" | "项目组做的会议文件能不能过内核" |
| 输入 | 会议基本信息（时间 / 地点 / 议案清单 / 出席人员等） + 项目标准目录 | 已起草的会议文件（单份或成套）+ 公司章程（如有） + 相关 DD Memo |
| 输出 | Markdown 初稿 + 一致性校验报告 | 带 tracked changes + comments 的 Word（`w:author="内核"`） |
| 修订方式 | 按模板生成 + 期限校验 | 标记偏差、提建议、不越俎代庖 |
| 典型触发语 | "起草股东大会文件" / "出一套开会的文件" | "内核" / "审阅" / "挑错" / "出批注" |

### 与 ecm-qc-shareholders-meeting-witness（见证业务）

| 维度 | ecm-qc-shareholders-meeting-witness | ecm-qc-meeting-docs-review（本 skill） |
|------|-----------------------------------|----------------------------------|
| 审查对象 | 律所出具的**《股东（大）会法律见证意见书》**（见证业务文书） | 公司自己出具的 **会议文件成套**（通知 / 议案 / 决议 / 记录等） |
| 视角 | "见证律师写的法律意见书有没有问题" | "公司发的会议文件有没有问题" |
| 业务场景 | 上市公司 / 新三板挂牌公司股东大会见证业务 | 所有公司的董监事会 / 股东会 / 股东大会会议文件（不限于上市公司） |
| 典型使用方 | 律所见证团队 + 内核 | 公司法务 + 律所项目组 + 内核 |

两者的交叉比对矩阵可**部分复用**（如"日期 / 届次 / 议案 / 特别决议"等核心字段），但视角不同：见证业务是"律师写对了吗"，本 skill 是"公司文件做对了吗"。

## 免责声明

本 skill 产出的修订意见和批注**不构成法律意见**，不替代签字律师 / 公司法务顾问的专业判断。完整免责声明见本仓库顶层 [DISCLAIMER.md](../../DISCLAIMER.md)。

## 配置项

**修订者名称（Reviewer Name）**

所有 `<w:ins>` / `<w:del>` / `<w:comment>` 的 `w:author` 默认 **`内核`**。用户可在对话开始时覆盖。

**审查深度（Review Depth）**
- `form-only`：仅形式审查（字体字号 / 文件结构 / 通知期限）
- `standard`（默认）：形式 + 实质 + 跨文件一致性
- `deep`：含 standard + 常见错误库扫描 + 会议记录逐段审

**审查范围（Review Scope）**
- `single`：用户只提交了一份文件（如只审通知 / 只审决议）；只做该文件自身的形式 + 实质审查，跳过跨文件一致性（也可让用户补上其他文件后重跑）
- `full-set`（默认）：成套审（通知 + 议案 + 决议 + 记录 + 签到表 + …）；执行跨文件一致性校验
- `post-meeting`：会后审（决议 / 记录 / 表决统计已填好实际数据）；额外校验表决票数基数、通过 / 否决判断

**公司性质（Company Nature）**

影响审查逻辑：
- `上市公司`：适用《上市公司股东会规则》、中小投资者单独计票、网络投票规则、交易所特殊规定
- `新三板挂牌`：适用全国股转系统规则
- `非上市股份公司`：《公司法》基本规则
- `有限责任公司`：使用"股东会"（非"股东大会"），章程约定优先

## 工作流（五步）

### Step 1 — 定位和预读入文件

```bash
ls /mnt/user-data/uploads/
```

识别用户提交的文件类型：
- **成套**（full-set）：多个文件，含通知 + 议案 + 决议 / 记录 / 签到表等
- **单份**（single）：仅一份（如只有通知 / 只有决议 / 只有记录）
- **会后审**（post-meeting）：文件中表决结果已填（不是空白 / 待填）

识别会议信息：
- 会议类型（股东会 / 股东大会 / 董事会 / 监事会）
- 会议属性（年会 / 临时会）
- 会议届次（董事会第 X 届第 Y 次 / YYYY 年第 N 次临时股东大会 / YYYY 年度股东大会）
- 召开日期 / 地点 / 召集人 / 主持人
- 议案清单（含特别决议 / 关联议案 / 中小投资者单独计票标注）

通读所有文件，提取：
- 公司全称 / 简称
- 是否上市公司 / 新三板挂牌 / 非上市股份 / 有限责任
- 总股本 / 总表决权股份（如为股份公司）
- 通知出具日期

### Step 1.5 — 拉取参考坐标（三级降级）

参考坐标按以下优先级获取：

1. **公司章程**（决定通知期限 / 表决比例 / 召开方式等"章程约定优先"事项）
2. **本套会议中的其他文件**（互为参考源 —— 起草人可能只给了一份，但内核可按项目目录找到其他文件）
3. **同项目 `DD-Memo-公司章程与治理` / `DD-Memo-批准和授权`**（用于会议议案合规性核对）
4. **同项目其他届次的会议文件**（用于届次号连续性校对）

#### Level 1 — 自动读取

```bash
# 公司章程
ls "${PROJECT_ROOT}/01-项目基础信息/" | grep -i "章程\|charter"

# 本套会议其他文件
ls "${PROJECT_ROOT}/04-文件输出/会议文件/{会议识别符}/"

# DD Memo
cat "${PROJECT_ROOT}/02-尽职调查/02-10-公司治理/DD-Memo-*.md"
cat "${PROJECT_ROOT}/02-尽职调查/02-01-批准和授权/DD-Memo-*.md"
```

#### Level 2 — 提示用户上传

Level 1 失败时：

> 【内核提示】自动从本项目目录读取参考文件未成功。为了做跨文件一致性和章程合规性交叉比对，请上传：
>
> - 公司章程（首选；决定通知期限 / 表决比例等）
> - 本次会议的其他相关文件（通知 / 议案 / 决议 / 记录 / 签到表等）
>
> 上传后我会继续审查。如无法提供，请回复"跳过比对"。

#### Level 3 — 跳过交叉比对，显式告知

在输出文档开头插入整体性批注：

> 【内核综合意见】本次内核审查**未进行与公司章程、本套其他会议文件的交叉核对**（参考文件未能获取 / 提供）。因此：
> - 通知期限 / 表决比例的章程约定 **未经外部文件核对**（退回到法定最低标准）
> - 跨文件一致性（议案清单 / 股权登记日 / 总股本等）**未经外部文件核对**
> - 本次批注仅基于本文件文本自身的形式审查和实质审查完成
>
> 如需完整内核，请补充公司章程或本套其他会议文件后重新提交。

### Step 2 — 三维度审查（同步记录 issues）

**按优先级顺序过一遍**：

0. **跨文件一致性交叉比对**（full-set 模式且 Step 1.5 成功时执行）— 阅读 [`references/cross-check-matrix.md`](./references/cross-check-matrix.md)，按 10 项跨文件字段核对：
   - 议案清单一致（通知 / 议案文本集 / 决议 / 表决统计表的议案编号 / 名称 / 顺序完全一致）
   - 股权登记日一致（通知 / 签到表 / 授权委托书）
   - 总股本 / 总表决权股份一致（通知 / 签到表 / 决议）
   - 会议召开基本情况一致（时间 / 地点 / 召集人 / 主持人 跨文件一致）
   - 特别决议标注一致（议案文本 / 决议 "已获 2/3 以上通过"）
   - 关联股东 / 董事回避标注一致（议案 / 决议 / 表决统计）
   - 中小投资者单独计票议案标注一致
   - 见证律师签字（如会议有见证）
   - 公司性质一致（上市 / 新三板 / 非上市股份 / 有限责任 —— 影响套用法规）
   - 章程依据一致（引用章程条款与公司章程一致）

1. **形式审查** — 阅读 [`references/form-requirements.md`](./references/form-requirements.md)，按 11 项清单逐条勾对：
   - 通知期限（硬校验；`meeting-docs-format §2.3`）
   - 通知的文件结构（6 段必备）
   - 议案的文件结构（4 段必备）
   - 决议的文件结构
   - 会议记录的文件结构（主持人 / 唱票人 / 监票人 / 记录人 / 见证律师签字）
   - 签到表的列定义
   - 授权委托书的文件结构（股份公司 / 上市公司）
   - 表决统计表的列定义
   - 代理投票计票细则（上市公司必备）
   - 字体字号（meeting-docs-format §八）
   - 日期 / 数字格式（meeting-docs-format §九）

2. **实质审查** — 阅读 [`references/substantive-checklist.md`](./references/substantive-checklist.md)，按 10 类审查项逐项过：
   - 召集权限合规（董事会 / 监事会 / 10% 以上股东自行召集）
   - 通知期限（法定 vs 章程）
   - 特别决议识别（公司合并分立 / 修章 / 增减资 / 发行公司债 / 章程特别约定等）
   - 关联股东 / 董事回避识别（关联交易 / 关联人提名董监高 / 关联方担保借款等）
   - 中小投资者单独计票识别（上市公司特有）
   - 累积投票制（董事 / 监事选举）
   - 网络投票（上市公司）
   - 表决票数基数计算（"占总股本"vs"占出席表决权股份总数"）
   - 表决结果与通过 / 否决判断（post-meeting 模式）
   - 2024 新《公司法》新情况（监事会可由审计委员会替代；一人公司 / 国有独资公司特殊规则）

3. **常见错误扫描** — 阅读 [`references/common-errors.md`](./references/common-errors.md)，按 A-I 类扫一遍。

**跨字段一致性检查**：
- 届次编号连贯（如"第三届第五次董事会"前应有"第四次"）
- 日期格式一致（"2026 年 4 月 24 日" vs "2026-04-24" vs "二〇二六年四月二十四日"）
- 公司简称一致
- 股数 / 金额数字千分位一致

### Step 3 — 准备修订工作目录

```bash
cd /home/claude
cp /mnt/user-data/uploads/xxx.docx ./input.docx
python /mnt/skills/public/docx/scripts/office/unpack.py input.docx unpacked/
```

如果是成套审（多份 .docx），每份单独 unpack / 修订 / pack。

### Step 4 — 写入修订痕迹和批注

遵循 `qc-skill-template.md` 的三条硬契约。

**对会议文件内核的特有技巧**：

- **股数 / 比例基数计算错误**：tracked change 改数字 + 【必改】批注引用 `meeting-docs-format §三` 的"基数计算规则"
- **特别决议未标注**：【必改】批注要求补标注（使用 `meeting-docs-format §三` 的标准话术"本项议案为特别决议事项，已经出席本次股东（大）会的股东及股东代理人所持表决权的三分之二以上同意通过"）
- **关联股东回避未标注**：tracked change + 【必改】批注
- **通知期限不足**：通知顶部打 ⚠️ + 【必改】批注指出具体天数

### Step 5 — 打包、输出、呈递

```bash
python /mnt/skills/public/docx/scripts/office/pack.py \
  unpacked/ /mnt/user-data/outputs/reviewed_{文件类型}.docx \
  --original input.docx
```

成套审时，每份文件独立输出。总输出一份**内核综合意见**（Markdown 或 .docx 附加文件），汇总跨文件一致性问题。

输出文件命名：`{会议识别符}_{文件类型}_内核后_{YYYYMMDD}.docx`（例："2026 年度股东大会_会议决议_内核后_20260424.docx"）。

`present_files` 呈递 + 简短总结。

## 输出格式契约（三条硬性要求）

完全沿用 `qc-skill-template.md` 的 3 条：
1. **修订者 = "内核"**（用户可覆盖）
2. **最小显示改动原则**
3. **解释文字只进批注**

**批注分类前缀**：
- **【必改】** 通知期限不足 / 特别决议未标注 / 关联回避未执行 / 跨文件不一致 / 基数计算错
- **【核实】** 章程条款依赖（如"通知期限是否有章程约定") / 召集权限（需核实召集人资格）
- **【建议】** 体例优化 / 表述统一
- **【底稿】** 需落实底稿（签字 / 盖章 / 现场出席确认）
- **【风险提示】** 可能导致决议效力瑕疵（如通知期限不足可能被撤销）的问题

## 边界与谨慎处理

**本 skill 不做**：
- 不替项目组补议案内容、补表决结果（走【必改】批注）
- 不对议案内容的合法性做终局判断（例某项重组方案是否构成借壳、某关联交易价格是否公允 —— 需律所项目组或保荐人综合判断）
- 不修改项目组对召集程序合法性的结论（除非存在明显瑕疵 —— 如通知期限明确不足；有疑虑走【核实】）
- 不自行核查股东资格（名册真实性）/ 董监高任职情况 —— 走【核实】 / 【底稿】批注

**遇到章程约定与法定最低标准冲突**（章程通知期限短于法定）：以法定为准 + 【必改】批注说明；但如果章程约定更**严格**（如通知期 25 日 > 法定 20 日），以章程为准。

**post-meeting 模式特有**：
- 表决结果与 "通过 / 否决" 判断冲突 → 【必改】批注
- 实际出席股数 < 法定最低（如有限责任公司股东会需代表 1/2 表决权）→ 【必改】【风险提示】批注

## 常见误用 / FAQ

1. **"我只有一份会议决议，其他文件还没写，能用本 skill 吗？"**：可以（single 模式）。但跳过跨文件一致性校验。建议补齐全套后重跑。

2. **"会议还没开，通知和议案已经出了，本 skill 能审吗？"**：可以（full-set 模式，pre-meeting）。本 skill 会：（1）审通知期限是否符合规则 （2）审议案是否标注特别决议 / 关联回避 （3）跳过表决票数校验（留空白）。

3. **"会议开完了，决议和记录写完了，想做最终审查"**：用 `post-meeting` 模式。本 skill 额外做：（1）表决票数基数计算 （2）通过 / 否决 判断一致性 （3）实际出席是否满足法定最低。

4. **"本 skill 和 `ecm-qc-shareholders-meeting-witness` 都审股东大会文件，有什么区别？"**：
   - 本 skill 审 **公司自己出的** 通知 / 议案 / 决议 / 记录等文件（文件主体：公司 + 董事会）
   - `ecm-qc-shareholders-meeting-witness` 审 **律所出的** 见证法律意见书（文件主体：律所签字律师）
   - 两者互相引用但不冲突：公司会议文件可经本 skill 内核，见证意见可经见证审查 skill 内核

5. **"2024 新《公司法》把监事会改成选配了，本 skill 会提示吗？"**：会。审查时若发现公司设了审计委员会代替监事会但会议文件仍出"监事会会议通知"，会【建议】批注提示确认公司治理架构。
