# 起草人自查（本 skill）vs 内核审查（内核审查版）— 边界说明

本文件详细说明 `ecm-draft:disclosure-review`（本 skill，起草人自查）与 `ecm-qc:disclosure-review`（内核审查版，内核审查）的边界、工作流差异和协作关系。

---

## 核心区别一览

| 维度 | ecm-draft:disclosure-review（本 skill） | ecm-qc:disclosure-review |
|------|--------------------------------------|------------------------------------|
| **使用者** | 项目组起草人（或项目组审核人） | 内核 / QC 团队（独立于项目组） |
| **角色** | "作者自己读一遍" | "审稿人给作者提意见" |
| **视角** | "我写的有没有问题？我写的跟工作报告对得上吗？" | "这一稿交给我了，能不能过内核出门？" |
| **工作性质** | 预检、预防 | 把关、准入 |
| **输出形式** | Markdown 自查报告（提建议，不改原文） | 带 tracked changes + comments 的 Word（直接标注修订） |
| **修订痕迹 author** | 无（不改原文） | `内核`（可配置） |
| **典型触发词** | "自己过一遍"、"预审"、"送内核前先检查"、"自查" | "内核"、"审阅"、"挑错"、"能不能过" |
| **迭代频次** | 每次起草人改稿后 1 次 | 起草人定稿后 1 次 |
| **输入** | 信披文件 + 工作报告 + DD Memo | 信披文件 + 工作报告 + 法律意见书（全套文件） |
| **风格审查深度** | 夸大 / 避重就轻 / 措辞模糊等 7 类 | 上述 7 类 + 法律意见措辞是否得当 + 律师应发表意见未发表 |
| **事实核查深度** | 交叉比对工作报告、DD Memo | 除本 skill 的核查外，还对照底稿、原始文件、监管规则 |
| **结论力度** | 建议级（提示律师考虑） | 决定级（不修改不得送监管） |

---

## 工作流差异

### 本 skill（起草人自查）

```
信披初稿 → 起草人自调 → 调用本 skill → 生成自查报告 → 起草人按报告修改 →
（可多次迭代） → 起草人认为基本可以送内核 → 送 qc 版
```

### qc 版（内核审查）

```
起草人提交的定稿 → 内核调用 qc skill → 自动生成带修订痕迹的审查稿 → 
内核审稿人复核 / 调整 qc 生成的修订意见 → 返回项目组 → 
项目组按修订意见改稿 → 再送内核（直到过审核）
```

---

## 为什么拆成两个 skill 而不是一个

1. **角色混淆会导致质量下滑**：一个 skill 既做起草人视角、又做内核视角，会导致两者都做得不到位。律所实务中起草人和内核是两拨人，skill 应尊重这个分工。
2. **输出形式根本不同**：起草人自查要快速迭代、提建议；内核要对原文做修订痕迹。一个 skill 难以兼顾。
3. **触发词不同**：起草人说"自查"，内核说"审阅"。描述范围不重叠，避免误触发。
4. **心态不同**：起草人自查时倾向"还能写得更好"，内核审查时倾向"这能不能过"。结论措辞、建议强度完全不同。

---

## 两个 skill 的 checklist 的重叠部分

以下 checklist 两个 skill **共用**（主要是事实性 / 逻辑性问题，两侧都要查）：

- 信披文件与工作报告的事实不一致
- 重大遗漏（工作报告的高风险未披露）
- 结论与事实不符
- 引用过时法规 / 已废止法规
- 涉外项目的审批 / 备案披露遗漏
- 关联披露不充分
- 承诺事项披露不完整
- 数据格式不规范

---

## 两个 skill 的差异部分

### 仅本 skill（起草人自查）独有

- **版本 Diff 分析**：起草人可能多版本迭代，本 skill 做版本间的 Diff；内核一般看的是定稿，不做 Diff
- **快速出报告**：起草人需要在 1-2 分钟内看完结果，本 skill 报告结构相对紧凑；内核的 Word 修订稿可能需要 10 分钟以上审阅
- **自定义严格度**：本 skill 有 `strict` / `normal` / `loose` 三级；内核一般固定最严格

### 仅 qc 版（内核审查）独有

- **修订痕迹输出**：内核直接在原文改，输出 tracked changes
- **批注（comments）输出**：内核会在原文上打批注说明修订理由
- **修订者 author**：所有修订的 `w:author` 统一为"内核"（可配置为"质控"/"DC"/律所自定义名）
- **审查深度**：会核对工作报告、法律意见书、会议文件三方一致性，不只看信披文件
- **法律意见措辞审查**：内核会对律师意见措辞的强弱、保留 / 限定的用词做把关
- **不只看"应披露内容"，还看"应发表意见"**：内核会识别律师本应对某事项发表意见但未发表的情形

## 协作建议

对于同一份信披文件：

1. **起草人完成初稿**：按 skill-roadmap 调用 `ecm-draft:report-assembly` + `ecm-draft:opinion-letter` 起草，然后人工起草信披文件
2. **起草人自查**：调用本 skill，按自查报告改稿
3. **起草人送内核**：交付给内核团队
4. **内核审查**：内核调用 `ecm-qc:disclosure-review`
5. **内核复核**：内核审稿人调整 qc 生成的修订建议（必要时加批注 / 删批注）
6. **返回起草人**：内核把带修订的 Word 返回项目组
7. **起草人响应修订**：起草人接受 / 拒绝修订，有争议的与内核沟通
8. **定稿**：双方确认后定稿，附申报材料

## 边界的实务示例

### 示例 1：措辞模糊"基本符合"

- **本 skill**：Step 4 发现"基本符合 xx 规定"，标记 ⚠️ "措辞模糊"，建议改为"符合……"+ 瑕疵说明
- **qc 版**：Step 类似，在 Word 上划线"基本"、添加批注"不得用'基本符合'，应明确符合或有限定"，w:author="内核"

### 示例 2：遗漏披露高风险

- **本 skill**：Step 3 方向二识别到工作报告高风险事项 X 未在信披文件披露，标记 🔴 "重大遗漏"，建议在重大事项提示段增加披露
- **qc 版**：直接在重大事项提示段插入拟增补的披露文字（tracked changes 为新增），附批注说明依据是工作报告附件四第 N 行

### 示例 3：引用《公司法》2018 年版

- **本 skill**：Step 4 扫描发现"《公司法》（2018 年修订）"，标记 ⚠️ "引用过时法规"，建议改为"《公司法》（2023 年修订）"
- **qc 版**：直接划线"2018 年修订"、改为"2023 年修订"；批注"《公司法》2023 年修订版自 2024 年 7 月 1 日施行，本申报适用 2023 版"

## 常见混淆

1. **用户说"帮我审阅一下招股书"**：这是**内核**的语义（"审阅"=审查），应触发 qc 版而非本 skill
2. **用户说"自己先过一遍"**：这是**起草人**的语义，触发本 skill
3. **用户说"能不能过内核"**：这实际上是起草人想**预测**内核的意见，应触发本 skill（帮起草人预演内核视角），但同时告知用户"真正的内核请调用 ecm-qc:disclosure-review"
4. **用户说"按内核的意见修改"**：这是起草人**响应内核修订**的场景，本 skill 和 qc 版都不合适；应让起草人按内核 Word 的修订痕迹直接改稿

---

## 总结

两个 skill 是"**预防**"和"**把关**"的分工。起草人自查和内核审查在法律实务中是两个独立的质量关卡，本仓库的 skill 设计尊重这种分工，避免角色混淆。
