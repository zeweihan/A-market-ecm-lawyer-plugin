# ecm-draft-disclosure-review（项目组起草人自查）

## 定位与边界

本 skill 是**项目组律师**在**将信披文件送交内核审查之前**的起草人自查工具。本 skill **不是**内核独立审查，不输出带修订痕迹的 Word 文件。

两个关键区别：

| 维度 | 本 skill（ecm-draft:disclosure-review） | ecm-qc:disclosure-review（BATCH-09，规划中） |
|------|--------------------------------------|----------------------------------------|
| 使用者 | 项目组起草人 | 内核 / QC 团队 |
| 视角 | "我写的有没有错、跟工作报告对不对得上" | "这一稿交给我了，能不能过内核" |
| 输出 | 自查报告（Markdown） | 带 tracked changes + comments 的 Word 文件（w:author="内核"） |
| 修订方式 | 提建议，律师自己改 | 直接给出修订内容 |
| 典型触发语 | "自己先过一遍"、"预审"、"送内核前检查" | "这份招股书能过内核吗"、"内核"、"审阅"、"挑错" |
| 后续流程 | 律师按建议修改后再送内核 | 内核意见发回起草人修改 |

本 skill **负责**：
- 读取信披文件（招股书 / 重组报告书 / 权益变动报告书 / 收购报告书 / 上市公告书 / 募集说明书 /
  年报或半年报的"法律相关"章节）
- 与**同项目**的律师工作报告（`ecm-draft-report-assembly` 输出）和 / 或 DD Memo 做**双向交叉比对**：
  - 方向一：信披文件→工作报告（信披的每个法律事实都能在工作报告找到依据？）
  - 方向二：工作报告→信披文件（工作报告识别的重大事项、"高"级风险，信披文件有没有披露？）
- 生成自查清单：披露遗漏 / 披露不一致 / 披露夸大 / 披露不准确
- 版本 Diff（若提供历史版本）：识别版本之间的变更点、评估变更合理性

本 skill **不负责**：
- 信披文件的首次起草（起草人自己写，本 skill 只做事后自查）
- 工作报告拼接 → `ecm-draft-report-assembly`
- 法律意见书起草 → `ecm-draft-opinion-letter`
- 内核审查 → `ecm-qc-disclosure-review`（BATCH-09）
- Word 格式最终调整 → `ecm-draft-format-adjust`
- 非法律相关章节（如"管理层讨论与分析"中的财务分析、"募投项目"中的项目可行性分析）

## 免责声明

本 skill 产出的自查报告仅供起草人内部使用，不替代律所内核审查、不构成对信披文件合规性的最终结论。完整免责声明见 [DISCLAIMER.md](../../DISCLAIMER.md)。

## 前置依赖

- **信披文件本身**：用户提供的 Word / Markdown / PDF
- **比对基准**（至少一个，优先顺序）：
  1. 同项目的律师工作报告（首选，最全面）
  2. 同项目的法律意见书（次选）
  3. 同项目的 DD Memo 原始件（兜底）
- **项目类型**：IPO / 再融资 / 并购重组 / 权益变动 / 其他（决定适用哪份自查清单）

## 核心工作流（六步）

### Step 1：识别信披文件类型与法律相关章节

按项目类型识别信披文件的**法律相关章节清单**（见 [references/disclosure-chapter-map.md](./references/disclosure-chapter-map.md)）。本 skill 只对这些章节做自查。

**典型法律相关章节**：

| 信披文件类型 | 法律相关章节（示例） |
|-----------|------------------|
| 招股说明书 | 重大事项提示、第三节"发行人及本次发行"、第四节"发行人基本情况"（发行人基本情况、股权结构、董监高）、第五节"业务与技术"（业务资质、重大合同）、第八节"公司治理"、第九节"关联交易和同业竞争"、第十节"主要财产"、第十一节"董监高及其他核心人员"、第十三节"财务会计信息"中的"主要债权债务"、第十四节"募集资金运用"、第十五节"诉讼、仲裁、行政处罚"、第十七节"其他重要事项" |
| 重大资产重组报告书 | 重大事项提示、第三节"交易对方情况"、第四节"交易标的"（包括业务、资产、负债、合规）、第五节"发行股份情况"、第八节"业绩补偿"、第九节"同业竞争和关联交易"、第十节"董监高"、第十一节"诉讼仲裁"、第十三节"其他重要事项" |
| 简式 / 详式权益变动报告书 | 收购人基本情况、权益变动目的、信息披露义务人前 6 个月买卖上市公司股份情况、附件（承诺函、授权委托书等） |
| 收购报告书 | 收购人基本情况、收购决定与收购目的、收购方式、要约收购条款、资金来源证明、后续计划、对被收购方的影响 |
| 上市公告书 | 发行人基本情况、本次发行情况、上市概况、股东及实际控制人、董监高、重要事项提示 |
| 募集说明书（公司债） | 发行人基本情况、财务会计信息、募集资金运用、重要事项提示 |

### Step 2：解析信披文件

- Word / Markdown → 结构化文本（按章节组织）
- PDF → 调用 `pdf` 外部 skill 提取文本 + 表格
- 按 Step 1 清单，只保留法律相关章节的文本
- 建立"章节 / 段落 / 页码"三级索引（供 Step 5 生成带页码的建议）

### Step 3：双向交叉比对

#### 方向一：信披文件 → 工作报告（"你写的对不对"）

对信披文件法律相关章节中的**每一项具体披露**（日期、股数、比例、金额、决议届次、事件经过、结论性表述等），在工作报告中搜索对应依据：

| 情形 | 标记 |
|------|------|
| 信披文件的表述在工作报告中能找到对应依据且文字一致 | ✅ 一致 |
| 信披文件的表述在工作报告中能找到对应依据但文字不同（同一事实不同措辞） | ⚠️ 表述差异 |
| 信披文件的表述在工作报告中找不到依据 | ❌ 无依据（可能是新添加 or 工作报告未覆盖） |
| 信披文件的表述与工作报告直接冲突（数据不一致、日期不一致、定性不一致） | 🔴 冲突（重点关注） |

#### 方向二：工作报告 → 信披文件（"你有没有漏披露"）

对工作报告中的**每一项重要事项**（全项目风险汇总表的全部条目、各章 Memo 的"总体结论"和"待补正事项"），在信披文件中搜索是否已披露：

| 情形 | 标记 |
|------|------|
| 工作报告的事项在信披文件中有披露 | ✅ 已披露 |
| 工作报告的"高"级风险在信披文件中未披露或披露不充分 | 🔴 重大遗漏 |
| 工作报告的"中"级风险在信披文件中未披露 | ⚠️ 建议披露 |
| 工作报告的"低"级风险在信披文件中未披露 | ℹ️ 可选披露 |

### Step 4：披露风格审查（独立维度）

对法律相关章节的文字表达做风格审查，识别以下问题（见 [references/disclosure-style-checklist.md](./references/disclosure-style-checklist.md)）：

- **夸大表述**："国内领先"、"行业第一"、"技术领先"等无客观依据的宣传语
- **避重就轻**：风险披露段过于笼统，未具体说明风险情形、程度、影响
- **措辞模糊**："基本符合"、"大致遵循"、"相关规定"（信披文件禁用）
- **结论与事实不符**：结论性表述（"不存在重大风险"）与详情段存在的具体事实矛盾
- **未定义缩写**：首次出现的缩写未在前文或释义段定义
- **引用过时法规**：引用已废止 / 已被修订的法规

### Step 5：输出自查报告

报告文件落到 `04-文件输出/信息披露文件/自查报告-{信披文件名}-{YYYYMMDD}.md`，结构：

```markdown
# {信披文件类型} 法律相关章节 起草人自查报告

> 项目：{company_short_name} | 信披文件版本：{V?.?} | 自查日期：YYYY-MM-DD | 自查人：{律师姓名} | 比对基准：{工作报告 / DD Memo} | skill 版本：{本 skill 版本}

## 一、自查范围

- 信披文件类型：{招股书 / 重组报告书 / ...}
- 法律相关章节清单：{本次自查覆盖的章节（对照 disclosure-chapter-map.md）}
- 比对基准文件：{工作报告路径} 或 {DD Memo 清单}
- 历史版本（Diff）：{无 / 提供了 vX.Y→vX.Z 的 diff}

## 二、冲突事项（🔴 重点）

（每条独立列示，不超过 1 屏）

### 冲突 1：{简短标题}
- **信披文件**：第 X 节第 Y 段，原文"……"
- **工作报告**：第 N 部分第 M 段，原文"……"
- **冲突类型**：数据不一致 / 日期不一致 / 定性不一致
- **建议**：以 {工作报告 / 信披文件} 为准，修改为"……"

## 三、重大遗漏事项（🔴）

### 遗漏 1：{简短标题}
- **工作报告识别的事项**：{全项目风险汇总表第 K 行；级别：高}
- **信披文件对应章节**：第 X 节（应披露但未披露）
- **建议**：在第 X 节 / 重大事项提示段增加以下披露："……"

## 四、表述差异事项（⚠️）

### 差异 1：{简短标题}
- **信披文件表述**：……
- **工作报告表述**：……
- **建议**：统一为"……"；或确认两种表述均准确，无需修改

## 五、披露风格问题（⚠️）

### 问题 1：{简短标题}（对照 disclosure-style-checklist.md 第 N 条）
- **位置**：第 X 节第 Y 段
- **原文**："……"
- **问题**：夸大表述 / 避重就轻 / 措辞模糊 / 结论与事实不符 / 未定义缩写 / 引用过时法规
- **建议改写**："……"

## 六、版本 Diff 摘要（若提供历史版本）

- 本版新增法律相关段落：{N} 处
- 本版删除法律相关段落：{M} 处
- 本版重大修改：{详情}
- 修改合理性评估：{合理 / 存在疑问，需关注}

## 七、自查结论

- 信披文件法律相关章节**基本**可送内核 / **需修改后再送内核** / **需重大修改**
- 必改项清单：{冲突事项 + 重大遗漏合集}
- 建议改项清单：{表述差异 + 披露风格合集}

## 八、后续流程建议

- 修改后可调用 `ecm-qc:disclosure-review`（BATCH-09）做内核独立审查
- 如涉及法规引用更新，可调用 `ecm-research:reg-search` 核验法规时效性
- 如涉及法律意见书同步修改，需同步更新 `ecm-draft:opinion-letter` 输出
```

### Step 6：交付与后续指引

- 自查报告 Markdown 落到上述路径
- 在报告末尾提示用户：
  - 按"必改项清单"先修改信披文件
  - 修改后可调用 BATCH-09 的 `ecm-qc:disclosure-review` 做内核审查
  - 若涉及工作报告修改，需协调起草人和 report-assembly 同步更新

## 配置项

### 比对严格度（Compare Strictness）

- `strict`：文本差异 > 5% 即标记（适合申报阶段最后一版）
- `normal`（默认）：文本差异 > 15% 标记（适合反馈阶段的迭代版）
- `loose`：只标记数据冲突和事实冲突（适合早期版本）

### 风格审查开关（Style Check）

- `on`（默认）：执行 Step 4
- `off`：跳过 Step 4（只做交叉比对，快速出自查报告）

### 版本 Diff 开关（Version Diff）

- `auto`（默认）：检测 `04-文件输出/信息披露文件/` 下的历史版本，自动做 Diff
- `off`：不做 Diff（首次自查或重写稿）
- `manual`：用户指定对照版本

## 输出格式契约

1. **文件路径**：`04-文件输出/信息披露文件/自查报告-{信披文件名}-{YYYYMMDD}.md`
2. **骨架**：8 段式（自查范围 / 冲突 / 重大遗漏 / 表述差异 / 披露风格 / 版本 Diff / 自查结论 / 后续流程）
3. **级别标记**：🔴 必改 / ⚠️ 建议改 / ℹ️ 可选改
4. **位置标记**：每条问题必须带"章节 / 段落 / 页码"
5. **不得**直接修改原信披文件（那是 `ecm-qc:disclosure-review` 或起草人手工的事）

## 与邻近 skill 的边界

- 与 `ecm-qc-disclosure-review`（BATCH-09）：本 skill 是自查，不输出修订 Word；qc 版是内核审查，输出带 tracked changes 的 Word。二者的 checklist 可复用，但视角和输出完全不同
- 与 `ecm-draft-report-assembly` / `opinion-letter`：本 skill 依赖前两者的输出作为比对基准
- 与 `ecm-research-reg-search`：本 skill Step 4 "引用过时法规" 检查可回调 research skill 核验时效

## 常见误用 / FAQ

1. **"招股书已经交到保荐人那边了，能用本 skill 做自查吗？"**：可以，但建议在交出去**之前**跑一遍。交出去后再自查，如发现问题要撤回改动，成本很高。
2. **"能不能直接改信披文件，不生成自查报告？"**：不能。本 skill 只提建议，不改原文件。直接改需要起草人自己操作（或等 BATCH-09 的 qc 版出现）。
3. **"工作报告还没出，能用本 skill 吗？"**：只能做风格审查（Step 4），不能做交叉比对。建议先调用 `ecm-draft:report-assembly` 拼工作报告后再跑本 skill。
4. **"信披文件是 PDF，本 skill 能读吗？"**：可以，会委托 `pdf` 外部 skill 提取文本 + 表格。但 PDF 复杂排版可能导致提取不全（图片、扫描页），此时提示用户提供 Word 源文件。
5. **"自查报告能直接给内核吗？"**：不建议直接给。自查报告是起草人内部工具，应按报告建议修改后再送内核；送内核的是**修改后的信披文件**，不是自查报告本身。

## 变更规则

- 输出契约变动 → MAJOR
- 信披文件类型扩展 / 章节映射表更新 → MINOR
- 披露风格清单补充 / 表述模板调整 → PATCH
