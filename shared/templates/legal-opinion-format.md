# 法律意见书排版规范（Single Source of Truth）

本文件是**唯一权威**的中国 A 股证券律师"法律意见书"（含首次公开发行、再融资、并购重组、重大资产重组等项目的法律意见书）文档结构与排版定义。

本规范与 `work-report-format.md` 配套——法律意见书和律师工作报告**形式配套**，签署律师 / 律所 / 编号 / 日期必须一致。

---

## 适用范围

- `ecm-draft:opinion-letter` 输出的主要类型：
  - 首次公开发行并上市之法律意见书（正式版 / 补充版）
  - 再融资（定增 / 配股 / 可转债）法律意见书
  - 并购重组 / 重大资产重组法律意见书
  - 股份发行 / 回购 / 减资等特定事项法律意见书
- `ecm-qc:opinion-letter-review`（BATCH-09）对项目组提交的意见书做格式偏离检核

**不适用**于：
- 律师工作报告 → `work-report-format.md`
- 股东（大）会见证意见 → 由 `ecm-qc:shareholders-meeting-witness` 按独立规范处理
- 备忘录 / 内部咨询意见 → `legal-memo-format.md`

---

## 一、意见书与工作报告的"形式配套"要求

律师工作报告是**"过程"文书**（描述律师做了哪些核查工作及发现），法律意见书是**"结论"文书**（基于工作报告发表意见）。两者必须满足：

| 一致性要求 | 强制级别 |
|-----------|---------|
| 律所名称、地址、联系方式完全一致 | 强制 |
| 签字律师完全一致（姓名、人数、顺序） | 强制 |
| 出具日期一致（同一天） | 强制 |
| 文号编制规则一致（如 XX 律字 [2026] 第 XXX 号 / 第 XXX-1 号） | 强制 |
| 编制依据（法律法规清单）一致 | 强制 |
| 核查范围描述一致（措辞相同或高度接近） | 强烈建议 |
| 结论性意见的内容不得超出工作报告所载发现 | 强制 |
| 所有"特别事项提示" / "保留意见"须与工作报告"三、风险分级汇总"的"高"级事项一致 | 强制（opinion-letter 自动注入时保证此点） |

## 二、首页封面

```
{律所全称（中文，三号宋体加粗）}
{律所全称（英文，小四 Times New Roman，斜体）}

（关于……的）
{律师事务所名称}
{法律意见书}
（主标题与律所名行，二号黑体加粗，段前段后各 36 磅）

致：{客户全称}

{律所文号}

{YYYY 年 MM 月 DD 日}
```

法律意见书的首页**不同**于工作报告的关键点：

- 主标题形如"XX 律师事务所关于 {发行人} {项目名} 之法律意见书"
- 如为补充 / 第 N 次意见书，需在文号处注明"第 XXX-1 号 / -2 号"并在标题加"之补充法律意见书"

## 三、正文骨架（强制 5 段）

法律意见书的正文骨架是**强制性的 5 段结构**，不得增删、改序：

```
一、引言

二、释义

三、正文（分章节发表意见）

四、结论性意见

五、保留 / 限定意见与特别事项提示（如有）
```

### 3.1 "一、引言" 段

包含以下 5 个小节：

```
（一）律师事务所简况（律所资质、执业许可证编号）
（二）经办律师简况（姓名、执业证号）
（三）本所出具法律意见书的身份与授权依据
（四）本次发行 / 交易概况
（五）核查范围与编制依据
```

### 3.2 "二、释义" 段

列示本意见书内全部专用名词的释义（如"本所"/"本次发行"/"发行人"/"保荐人"/"报告期"等）。每条独立列示：

```markdown
**本所** 指 {律所全称}
**本次发行** 指 {项目名称}
**发行人** 指 {客户全称}
```

释义数量一般 15-30 条。

### 3.3 "三、正文"段（按编报规则 17 章对应发表意见）

正文部分 **1 章对应 1 个意见小节**，章节名与《编报规则第 12 号》一致：

```
（一）关于本次发行的批准和授权
（二）关于发行人的主体资格
（三）关于发行人的设立
（四）关于发行人的独立性
……
（十七）关于其他合规事项
```

每个小节内部遵循"**事实陈述 → 核查结论 → 法律意见**"三步法：

```markdown
（一）关于本次发行的批准和授权

1、事实陈述
（引述董事会 / 股东大会决议内容、外部批文取得情况）

2、核查工作
（简述核查的资料范围与方法）

3、法律意见
经核查，本所认为：
（1）发行人本次发行已经履行必要的内部审议程序；
（2）……；
（3）综上，本所认为，发行人本次发行的批准和授权合法有效，符合《公司法》《证券法》《首次公开发行股票注册管理办法》的有关规定。
```

### 3.4 "四、结论性意见" 段

一段**总括意见**，格式固定：

```
综上所述，本所认为：发行人本次发行符合《中华人民共和国公司法》《中华人民共和国证券法》《首次公开发行股票注册管理办法》等相关法律、法规、规范性文件关于首次公开发行股票并上市的实质条件；本法律意见书所涉及事项如有变化，本所将另行出具补充法律意见。

（下附特别事项提示，如有）
```

### 3.5 "五、保留 / 限定意见与特别事项提示" 段（如有）

opinion-letter 从 DD Memo 的 `dd-output-schema.md` 定义的"风险分级汇总"表中抽取 `级别=高` 事项注入本段，每条格式：

```markdown
### 特别事项 1：{章节名}

经核查，发行人 {问题简述}。
法规依据：{法规条文}。
本所意见：{是否构成发行障碍；是否需整改 / 补充披露}。
```

## 五、编号规则

| 层级 | 编号格式 | 示例 |
|------|---------|------|
| 主章 | 汉字序数 `一、二、` | `一、引言` |
| 主章下的子段 | 括号汉字 `（一）（二）` | `（一）律师事务所简况` |
| 小节 | 阿拉伯数字 + 顿号 `1、2、` | `1、事实陈述` |
| 列举 | 括号阿拉伯数字 `（1）（2）` | `（1）发行人本次发行……` |
| 再细列举 | 带圈数字 `①②` | `① 独立董事事前认可……` |

## 六、语言风格（与工作报告一致）

- 第一人称统一为"本所"
- 禁用"显然""明显""必然"等主观化措辞
- 结论用 `合法有效 / 符合……的规定` / `不存在需要披露的违法违规情形`
- 保留 / 限定用 `除……外 / 在……前提下`

---

## 七、页眉页脚 / 页码

- 页眉奇数页右对齐"法律意见书"，偶数页左对齐律所名
- 页脚中央"第 X 页，共 Y 页"
- 封面、释义页不显示页码；从引言起算页码

---

## 八、末页签字盖章

```
{律所全称}（盖章）：

{律所负责人姓名}（签字）：_____________________

经办律师（签字）：
    {律师 A 姓名}：_____________________
    {律师 B 姓名}：_____________________

{YYYY 年 MM 月 DD 日}
```

**强制**：律所负责人签字；至少 2 名经办律师；日期与封面一致。

## 九、opinion-letter 自动填充指南（供 Claude 查表）

opinion-letter 运行时按下列顺序填充本模板：

1. **封面**：从项目元信息（见 `dd-output-schema.md` §1）读 `company_short_name`、`lawyer_name`；日期用"今天"
2. **释义段**：默认 15 条基础释义；opinion-letter 扫描工作报告文字抽出额外专有名词，追加到释义表
3. **正文三步法"事实陈述"**：从对应 DD Memo "四、结论与建议"的"总体结论"段抽取
4. **正文三步法"核查工作"**：从对应 DD Memo "一、核查要点清单"段抽取（简化措辞）
5. **正文三步法"法律意见"**：结合 DD Memo 的"总体结论"+ 法规引用（来自 `shared/regulations/`），按"本所认为……"句式组织
6. **特别事项提示**：按 `dd-output-schema.md` §5 规则抽取 `级别=高` 事项
7. **结论性意见**：按模板 §3.4 固定措辞，插入项目类型（IPO / 再融资 / 并购）对应的法规清单

## 十、常见踩坑

1. **意见书结论超出工作报告认定范围** → QC 反馈时的高频问题 → opinion-letter 必须严格引用 DD Memo 的"结论与建议"段，不自行生成新事实
2. **律所名称 / 签字律师 / 日期与工作报告不一致** → 系统性合规瑕疵 → opinion-letter 运行时必须与同项目的 report-assembly 共享项目元信息，避免分别生成
3. **特别事项提示遗漏高级风险** → opinion-letter 必须遍历 17 份 Memo 的汇总表，不允许按章节选择性注入
4. **释义与工作报告不一致** → 两个文件共用一份释义源（opinion-letter 运行时从 report-assembly 输出中抽取）
5. **使用未定义缩写** → 所有缩写必须在"二、释义"段定义；opinion-letter 扫描正文出现的缩写，若未定义则自动提示
